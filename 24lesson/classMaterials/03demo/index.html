The Loop Trap

var does not create a new scope per loop iteration — there is one shared i that all pushed functions close over. By the time any of them runs, the loop has finished and i is already 3. let fixes this by creating a fresh binding per iteration.

broken — var (one shared binding)
var fns = [];

for (var i = 0; i < 3; i++) {
  fns.push(() => i);
}

// loop ends — i is now 3
// all three functions share
// that same i binding

fns[0]();  // → 3  (not 0!)
fns[1]();  // → 3
fns[2]();  // → 3
fixed — let (per-iteration scope)
let fns = [];

for (let i = 0; i < 3; i++) {
  fns.push(() => i);
}

// each iteration created its own i
// each function closed over
// a different scope instance

fns[0]();  // → 0
fns[1]();  // → 1
fns[2]();  // → 2

bonus — the pre-let IIFE workaround
before ES6, developers forced a new scope per iteration with an IIFE
var fns = [];

for (var i = 0; i < 3; i++) {
  fns.push(
    (function(captured) {    // IIFE creates a new scope
      return function() { return captured; };
    })(i)                       // i's current value passed as argument
  );
}

fns[0]();  // → 0   works — captured is a fresh binding each time

This pattern is worth showing because it proves the bug was always fixable via closures — let just made the engine do the IIFE bookkeeping for you automatically.

what to notice
  • var is function-scoped, not block-scoped. The loop body is not a new scope — i is hoisted to the enclosing function (or global) and mutated in place.
  • All three arrow functions close over the same i binding. When the loop finishes, that binding holds 3. Every call returns 3.
  • let is block-scoped. The spec requires a fresh i binding per iteration. Each pushed function closes over a distinct scope instance — different heap address, different value.
  • The IIFE workaround shows this was always a closures problem solvable with closures. let is syntactic sugar over the same underlying mechanism.
